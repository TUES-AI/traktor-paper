# Summary: PathNet: Learning To Generate Trajectories Avoiding Obstacles

**Source**: ICIP 2020 / IEEE SPS | **Authors**: Alassane Watt, Yusuke Yoshiyasu | **DOI**: 10.17023/0w6r-q127

Note: this paper does not appear to have an arXiv TeX source available. This summary uses the same project summary format as the arXiv summaries, based on the IEEE/SPS metadata and abstract-level technical description.

---

## Problem and Core Idea

PathNet solves 2D motion planning by directly generating a whole trajectory from a map, start point, and goal point. The planner takes a 2D environment map split into obstacle and free zones, compresses the map into a latent vector, concatenates that latent vector with the start and goal positions, and outputs a trajectory connecting them.

The key difference from MPNet is that PathNet is a **one-shot trajectory generator**. MPNet predicts the next waypoint repeatedly; PathNet predicts the full trajectory in a single forward pass.

---

## Method Details

- **Input**: 2D map with free/obstacle cells, start position, goal position.
- **Environment encoder**: compresses the 2D map into a latent vector representing obstacle layout.
- **Fusion**: concatenates environment latent with start and goal coordinates.
- **Trajectory generator**: decodes the fused vector into a sequence of trajectory points.
- **Output**: full planned 2D trajectory from start to goal.
- **Training style**: supervised learning from valid example trajectories generated by a planner or dataset.
- **Inference**: one network pass; no recursive expansion required.

A practical rover version should be:

```text
64x64 occupancy grid
    -> small CNN encoder
    -> latent z
concat(z, start_xy, goal_xy)
    -> MLP decoder
    -> K waypoints, shape K x 2
```

Recommended small architecture:

```text
CNN encoder:
- Conv2d 1 -> 16, kernel 5, stride 2, ReLU
- Conv2d 16 -> 32, kernel 3, stride 2, ReLU
- Conv2d 32 -> 64, kernel 3, stride 2, ReLU
- flatten
- Linear -> 128 latent

Trajectory head:
- concat latent + start_xy + goal_xy
- Linear 132 -> 256, ReLU
- Linear 256 -> 256, ReLU
- Linear 256 -> 2K

Output:
- K fixed waypoints, e.g. K = 32
```

Useful losses for this project:

```text
L = waypoint_mse
  + endpoint_penalty
  + obstacle_collision_penalty
  + smoothness_penalty
```

## Key Results

- The paper claims PathNet solves 2D motion-planning problems in unseen environments.
- The main efficiency claim is one-pass trajectory generation: a complete trajectory is produced in a single network forward pass.
- Because it generates the full path directly, it can be faster than search or recursive neural planners at inference time.
- The abstract-level description emphasizes obstacle avoidance and generalization, but detailed quantitative results were not available from arXiv TeX because the paper is not arXiv-sourced.

---

## Relevance to This Project

- **Direct trajectory constructor**: PathNet matches the exact need for constructing 2D rover trajectories rather than only choosing actions.
- **High-level planner candidate**: it can serve as the slower/global network in the two-network project idea.
- **Pi-friendly inference**: one forward pass over a small occupancy grid and MLP decoder should be easy to run on Raspberry Pi.
- **Good paper framing**: PathNet gives a clean baseline for learned trajectory construction, while the project novelty can focus on adaptive trajectory execution under changing terrain.
- **Coverage path planning**: PathNet can connect coverage waypoints or frontier goals, but it does not by itself decide which areas to cover next.
- **Compatibility with PPO**: PPO or another low-level controller can track the generated waypoint sequence.

## Concrete Experiments to Run

1. Generate A* or RRT* paths in `rover_coverage_env.py` and resample each path to a fixed `K=32` waypoints.
2. Train a small PathNet-style CNN+MLP to predict the full waypoint sequence from occupancy grid, start, and goal.
3. Compare against MPNet-style recursive waypoint prediction on inference speed, collision rate, path smoothness, and robustness to unseen obstacle layouts.
4. Add collision-check repair: if any predicted segment crosses an obstacle, repair only that segment with A* or RRT*.
5. Feed generated paths to a low-level rover tracker and evaluate coverage progress versus direct reactive policies.
6. Test whether the same trained planner transfers from sim occupancy maps to maps built from ultrasonic/camera data.

---

## Risks / Open Questions

- One-shot trajectory generation is brittle: a single bad point can make the whole path invalid.
- Fixed-length trajectory output requires resampling paths; this can distort paths with very different lengths or topology.
- The model may average over multiple valid homotopy classes and generate paths that cut through obstacles unless collision penalties or repair are added.
- It has no built-in completeness guarantee, unlike MPNet with hybrid RRT* repair.
- It assumes a map is available. The physical rover currently has sparse range sensing, so mapping/local occupancy construction is needed.
- It plans geometric paths, not dynamically feasible differential-drive trajectories; a tracker or curvature-aware postprocessor is required.
