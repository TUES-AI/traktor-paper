# Summary: Motion Planning Networks

**arXiv**: 1806.05767 | **Authors**: Ahmed H. Qureshi, Anthony Simeonov, Mayur J. Bency, Michael C. Yip | **Venue**: ICRA 2019

---

## Problem and Core Idea

Classical sampling-based planners such as RRT*, Informed-RRT*, and BIT* can produce collision-free paths, but planning time grows badly with problem dimensionality and clutter. MPNet learns a motion-planning prior from expert demonstrations: encode the obstacle space into a latent vector, then use a neural planning network to iteratively predict the next robot configuration from the current configuration toward the goal.

The key idea is not to replace collision checking entirely. MPNet uses the network to generate fast candidate paths, then verifies and repairs them with steering, lazy state contraction, neural replanning, or hybrid replanning with a classical planner.

---

## Method Details

- **Input**: obstacle representation as point cloud or workspace vector, current/start configuration, goal configuration.
- **Encoder network (Enet)**: maps obstacle point cloud `x_obs` to latent vector `Z`. In the paper this can be trained as a contractive autoencoder or end-to-end with the planner.
- **Planning network (Pnet)**: feed-forward MLP that predicts the next state: `x_{t+1} = Pnet(Z, x_t, x_goal)`.
- **Training signal**: supervised imitation from expert paths generated by RRT* or another planner. Loss is MSE between predicted next state and expert next state.
- **Online planning**: bidirectional neural planner grows one path from start and one from goal, alternately extending each side until `steerTo` says the two sides are connectable.
- **Path cleanup**: lazy state contraction removes unnecessary intermediate states by shortcutting collision-free straight-line segments.
- **Repair**: if path segments are infeasible, neural replanning recursively repairs them; hybrid replanning falls back to RRT* for completeness.
- **Stochasticity**: dropout is kept active in hidden layers so repeated replanning attempts can produce different paths.

Implementation details from the linked repo:

- Pnet is a deep PReLU MLP with dropout and output dimension equal to the robot configuration dimension.
- Simple 2D example uses `MLP(32, 2)`: environment encoding plus start/goal state to next 2D state.
- CAE encoder example compresses a `2800`-dim obstacle vector to a `28`-dim latent.
- Training uses `MSELoss` and Adagrad over expert next-state targets.

## Key Results

- MPNet generalizes to unseen start/goal pairs in seen workspaces and also to unseen workspaces.
- Hybrid replanning reaches **100% success** in the reported 2D/3D environments while retaining large speedups.
- Neural-only replanning averages around **97% success** over reported trials.
- Planning time stays around or below **1 second** in the reported settings.
- Reported speedups are about **40x faster than Informed-RRT*** and **20x faster than BIT*** on average.
- In simple 2D, MPNet neural replanning takes about **0.11s** versus BIT* around **2.6-2.7s**.
- In complex 2D, MPNet neural replanning takes about **0.17-0.18s** versus BIT* around **3.8-4.1s**.
- On Baxter 7-DOF experiments, MPNet averages about **1s** planning with **85% success**, while BIT* has lower success under comparable path-length constraints.

## Relevance to This Project

- **High-level planner fit**: MPNet directly matches the project's need for a learned 2D trajectory planner: `map + current pose + goal -> next waypoint`.
- **Good pairing with local adaptation**: MPNet can be the slow/high-level planner, while the rover's small adaptive controller learns how to track each local segment under changing terrain.
- **Safety story**: the hybrid replanning design is important for real rover deployment because the neural planner is never trusted blindly; paths are collision-checked and repaired.
- **Pi constraints**: a small 2D CNN/MLP version should be feasible on Raspberry Pi, especially if the obstacle map is low-resolution and the output is only the next waypoint.
- **Coverage path planning**: MPNet is start-goal planning, not coverage planning, but it can be used as the local planner between coverage frontier goals or sweep waypoints.
- **PPO compatibility**: PPO can remain the low-level controller/tracker, while MPNet provides waypoint targets or planned path segments.

## Risks / Open Questions

- MPNet solves point-to-point motion planning, not full coverage. We still need a coverage-goal selector or sweep/frontier planner above it.
- The paper uses obstacle point clouds and ideal collision checking; our rover has sparse ultrasonic sensing unless we construct a map over time.
- Straight-line `steerTo` works for point-mass planning but differential-drive dynamics need curvature/turn-radius constraints or a separate trajectory tracker.
- Neural-only planning can produce invalid paths, so collision checking and repair are mandatory for physical tests.
- Training data generation may dominate setup time if many obstacle layouts and expert paths are needed.
- The original implementation is old PyTorch/Python 2 style; for this project we should reimplement the small 2D version cleanly rather than port it directly.
